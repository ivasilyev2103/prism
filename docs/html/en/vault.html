Vault — Secrets Storage

The only module that knows plaintext provider API keys

RU EN

Purpose

Secret storage implementing the Sign-not-Expose pattern. API keys are encrypted with AES-256-GCM per-record and never leave Vault in plaintext.

Vault is the single point through which all provider API key access occurs. No other module has access to plaintext key values.

Storage

SQLite + AES-256-GCM

Storage is built on modernc.org/sqlite (pure Go, no CGO). Each record is encrypted individually with AES-256-GCM. Storage format:

nonce (12 bytes) || ciphertext

Per-record encryption allows atomically updating individual keys without re-encrypting the entire database.

Key Derivation

Argon2id

Key derivation function parameters:

ParameterValue
Iterations3
Memory64 MB
Threads4
Key length32 bytes

Salt is stored openly in metadata. Derived encryption key resides in RAM only (mlock-pinned buffer).

Sign-not-Expose

Custom RoundTripper

The Sign-not-Expose pattern is implemented via a custom http.RoundTripper that injects the Authorization header at the transport level and zeroes it after the wire write.

The plaintext key never lingers in http.Request.Header. Calling code never gains access to the key value — only to the signed request.

Tokens

Project Authentication

ValidateToken() uses HMAC(token) for DB lookup + constant-time compare. Tokens are stored as HMAC hashes — plaintext is never in the DB.

TTL is configurable per project. When TTL expires, the token is automatically invalidated.

Key Rotation

Atomic Key Replacement

RotateProviderKey() atomically replaces the key in a single transaction. The old key is immediately deleted and zeroed. During rotation, requests to the provider are blocked until the transaction completes.

Memory Safety

explicit_bzero() is best-effort in Go. The GC may copy data before zeroing.

The real security boundary is process isolation + mlock. Prism binds only to 127.0.0.1, and mTLS provides protection at the process level.
