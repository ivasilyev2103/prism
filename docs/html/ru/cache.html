Cache — Семантический кэш

Кэширование похожих запросов с VP-tree индексом

RU EN

Назначение

Кэширует ответы на семантически похожие запросы. Эмбеддинги генерируются локально через Ollama (nomic-embed-text). VP-tree обеспечивает O(log n) поиск ближайших соседей.

Prism — универсальный AI-шлюз, поэтому кэш учитывает ServiceType: детерминистичные сервисы (chat, embedding) кэшируются по умолчанию, стохастичные (image, 3d_model) — нет.

Алгоритм

Поиск по cosine similarity

embed(text) → cosine_similarity с кэшем
    ↓
similarity > 0.95 → cache hit  → вернуть закэшированный ответ
similarity ≤ 0.95 → cache miss → запрос к провайдеру → сохранение в кэш

Threshold 0.95 — глобальное значение для всех проектов. Per-project настройка планируется в будущих фазах.

Cache Policy

ServiceType Кэширование TTL (с) Причина
chat Да 3600 Детерминистично при temperature=0
embedding Да 86400 Полностью детерминистично
image Нет — Стохастично
tts Настраиваемо 86400 Зависит от voice+text
3d_model Нет — Стохастично
stt Нет — Audio input
moderation Нет — Нет смысла

Get и Set первым делом проверяют CachePolicy для req.ServiceType. Если кэширование отключено: Get возвращает miss, Set — no-op.

Шифрование

Sanitized ответы + зашифрованные PII mappings

Кэш хранит два компонента для каждой записи:

  • Sanitized response — ответ с placeholder'ами вместо реальных PII
  • Encrypted PII mapping — зашифрованный per-entry (AES-256-GCM) маппинг placeholder → оригинальное значение

PII никогда не хранится на диске в plaintext. Каждая запись шифруется отдельным ключом (derived от master key + entry ID). Компрометация одной записи не раскрывает другие.

При cache hit:

  1. Найти семантически похожий запрос (cosine similarity > threshold)
  2. Расшифровать PII mapping для данной записи
  3. Применить restoration (подменить placeholder'ы на оригинальные данные)
  4. Вернуть восстановленный ответ

VP-Tree

In-memory vector index

Вместо brute-force O(n) scan по SQLite используется VP-tree (Vantage Point tree) для O(log n) lookup:

  • Embeddings загружаются в память при старте из SQLite
  • Index обновляется при добавлении новых записей (Set)
  • SQLite (modernc.org/sqlite, pure Go) — persistent backend для durability
  • При рестарте — index восстанавливается из SQLite

Деградация

Ollama недоступна

Если Ollama недоступна, Semantic Cache деградирует gracefully:

Метод Поведение
Get Возвращает miss (не ошибку)
Set No-op (не записывает)

Policy Engine продолжает работу без кэша. Приложение не знает о деградации — просто нет cache hit.

Интерфейс

type SemanticCache interface {
    // Get ищет семантически похожий запрос в кэше.
    // Возвращает (nil, nil) при промахе или если кэширование отключено.
    Get(ctx context.Context, req *types.SanitizedRequest) (*types.Response, error)

    // Set сохраняет запрос и ответ в кэш.
    // Для ServiceType с отключенным кэшированием — no-op.
    Set(ctx context.Context, req *types.SanitizedRequest, resp *types.Response) error

    // Invalidate удаляет записи по projectID.
    Invalidate(ctx context.Context, projectID string) error
}

type Embedder interface {
    // Embed возвращает embedding для текста.
    // Реализация: nomic-embed-text через Ollama (локально).
    Embed(ctx context.Context, text string) ([]float32, error)

    // HealthCheck проверяет доступность Ollama.
    HealthCheck(ctx context.Context) error
}

type CachePolicy struct {
    ServiceType types.ServiceType
    Enabled     bool
    TTL         int // секунды; 0 = default
}
